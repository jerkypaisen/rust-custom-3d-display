# Rust 無改造クライアントに独自3Dモデルを表示する実証実験

Rust（ゲーム）の**クライアントを一切改造せず**、サーバープラグインだけで
独自の3Dモデル（例: MMDのミク）を色付きでワールド上に表示する手法の実証コードです。

「描画系をサーバーから叩いてカスタムメッシュを出せないか」という問いに対する、
一連の検証・実装・そして到達した限界の記録です。結論を先に言うと:

- **静止した色付き3Dモデルの表示は実用レベルで可能**（`ddraw.line` のスキャンライン充填）
- **滑らかなアニメーションは原理的な壁にぶつかる**（後述）

> ⚠️ このリポジトリには Rust のデコンパイルソースや MMD モデル（ミク）等の
> 著作物は含みません。プラグイン・変換ツール・ドキュメントのみです。

---

## 仕組みの概要

```
3Dモデル(PMX/GLB/OBJ…)
   │  tools/model2mdraw.py   ← サーバー外でオフライン変換
   ▼
拡張OBJ (.obj) + テクスチャ埋め込み + スキン(.skin) + モーション(.anim)
   │  oxide/data/MeshDraw/ に配置
   ▼
plugin/MeshSurfaceDraw.cs  ← Oxide/Carbonプラグイン
   │  ddraw.line を大量送信して面を「線の充填」で塗る
   ▼
Rustクライアント（無改造）がワールド3D空間に描画
```

ポイントは2つ:

1. **オフライン変換**: 重い処理（テクスチャのベイク、ポリゴン削減、隠面除去、
   スキンウェイトの転写）は全てPython側で済ませ、プラグインは軽い描画に専念する。
2. **描画は `ddraw.line` のみ**: サーバーが admin プレイヤーのコンソールに
   `ddraw.line` コマンドを送ると、クライアントがワールド座標に線を引く。
   三角形を平行線で塗りつぶす（スキャンライン）ことで「面」を表現する。

## ddraw について判明した重要な性質（実験で確定）

開発中に行った検証実験で判明した事実（詳細は [docs/FINDINGS.md](docs/FINDINGS.md)）:

| 実験 | 内容 | 結果 |
|---|---|---|
| 線の前後関係 | 近い線(先送信)と遠い線(後送信)を交差表示 | **交点=後送信の色**。線同士は深度判定されず「後から送った線が常に上」 |
| 球エンティティ越しの線 | sphere.prefab越しに線を見る | 球は半透明で深度を書かない |
| 氷の彫刻 | サーバーからボクセル彫刻 | 彫刻は不透明で深度を書く。サーバーからリフレクションで形状制御可能 |

**最重要の結論**:
`ddraw.line` は地形・建物に対しては正しく深度テストされる（木の後ろに隠れる）が、
**線同士には深度がなく、描画順（後勝ち）で前後が決まる**。
このため、視点に応じて毎回「遠→近」の順でソートして送らないと前後関係が破綻する。

## できること / できないこと

### ✅ 静止表示（実用レベル）
- PMXのテクスチャ・複数マテリアルを「面ごとの代表色」または
  「線分割でのテクスチャサンプリング」で再現（顔の目・口まで）
- 画面ピクセル基準の適応密度・重ね描き・インナーコア（裏打ち）で隙間を低減
- 正面〜斜めからの観賞に最適（前後関係は描画時の固定視点基準）

### ❌ 滑らかなアニメーション（原理的な壁）
スキニング（PMXボーン+ウェイト、VMDモーションのキーフレーム補間）自体は実装でき、
**ポーズの計算は正しく動く**。しかし表示が破綻する:

1. **線同士に深度がない** → 部分更新（動く部位だけ再送）すると、再送した線が
   常に最前面に来て、背中側の髪が体を貫通して見える。
   → 毎フレーム全身を遠→近ソートで送り直すしかない。
2. **スループットの壁** → 1フレームに送れる `ddraw.line` の数には上限があり、
   それを超えると次フレームが間に合わず、線が寿命切れで点滅・消失する。
   実用的な密度で滑らかに動かすだけの帯域が出ない。

結論として、**「色付き・高精細・全方向ボリューム・滑らかアニメ」を同時に**
クライアント無改造で出すことは原理的に不可能。各手法は必ずどれか1軸を犠牲にする:

| 手法 | 色 | 立体(全方向) | 高精細 | 滑らかアニメ |
|---|---|---|---|---|
| ddraw.line | ○ | ○ | ○ | ✕(スループット) |
| サイン/ネオン画像 | ○ | ✕(平面) | ○ | △(フレーム送り) |
| 彫刻エンティティ | △(単色/パレット) | ○ | ✕(ボクセル) | ○(補間) |

## 使い方

### 1. 依存関係（変換ツール側 / Python 3.10+）
```bash
pip install -r tools/requirements.txt
```

### 2. モデルを変換（劣化なし＝推奨）

**`-f` を付けない**のが基本です。ポリゴンを一切削減しないので、形・テクスチャ・UVが
元モデルのまま保たれ、削減由来のアーティファクト（境界の亀裂・色の飛び・UV島の乱れ）が
一切出ません。**最も綺麗に出したいならこれ。**

```bash
# 劣化なし変換（推奨）。高さ1.6mに正規化してそのまま出力
python tools/model2mdraw.py model.pmx -H 1.6 -o miku.obj
```

入力は PMX / GLB / GLTF / OBJ / STL / PLY / 3MF（FBXは不可。Blenderで一度GLBへ）。
テクスチャは自動でアトラスに埋め込まれ、面ごとの色＋線分割サンプリングで再現されます。

### 3. 変換結果を確認（ゲームに入れる前のオフラインプレビュー）
```bash
python tools/preview_obj.py miku.obj out.png --yaw 0   # 静止プレビュー（プラグインと同じ描画を再現）
python tools/validate_obj.py miku.obj                  # 整合性チェック
```

### 4. サーバーへ配置
`miku.obj` を `oxide/data/MeshDraw/` にコピーし、
`plugin/MeshSurfaceDraw.cs` を `oxide/plugins/` に入れて読み込む。

### 5. ゲーム内で綺麗に描画する手順 ★ここが品質の肝

```
/mdraw miku 1            # まず表示（見せたい正面に立ってから実行＝その向きが正面基準になる）
```

劣化なし変換は面数が多い（数万面）ため、**線数の上限を上げないと薄く（ワイヤーフレーム気味に）なります。**
以下を順に上げて、ちょうど良い密度を探ってください（変更後は `/mdraw refresh` で反映）:

```
/mdraw lines 150000      # ① 線数上限を大きく（劣化なし高ポリでは必須。重ければ下げる）
/mdraw dist 0.5          # ② 想定観賞距離。小さいほど高密度（既定0.5）
/mdraw px 0.8            # ③ 線の画面間隔。小さいほど濃い（1px未満でベタ塗りに近づく）
/mdraw passes 3          # ④ 重ね描き層数。多いほど隙間が埋まる
/mdraw refresh           # 設定を反映して描き直し
```

うまく出るときの目安：**正面〜斜め45度**から観賞、`core`（裏打ち）ON のまま、
`shade` OFF（テクスチャの原色を活かす＝アニメ調モデル向き）。

仕上げの微調整:
- 近くで線が目立つ → `/mdraw dist 0.3` や `/mdraw px 0.6`（さらに高密度）
- 重い／クライアントFPSが落ちる → `/mdraw lines` を下げる（後述のTIPSで面数削減も検討）
- 輪郭が汚い → `/mdraw crease 35`（浅い折れ目を無視）、不要なら `/mdraw edge 0`

主なコマンドは [docs/COMMANDS.md](docs/COMMANDS.md) を参照。

## TIPS

### ポリゴン削減（劣化あり・軽量化したいとき）
`-f <面数>` を付けると目標面数まで削減します。**形が少し崩れたり、境界やUVに
わずかなアーティファクトが出ることがあります**が、面数が減るので少ない線数でも密に見え、
描画が軽くなります。重さと品質のトレードオフ用。

```bash
python tools/model2mdraw.py model.pmx -H 1.6 -f 8000 -o miku.obj   # 8000面に削減
```

削減は「全体を一体に溶接してから削減し、UV・材質・スキンウェイトを最寄り面から転写」する
方式なので、マテリアル境界の亀裂は出ません。それでも顔など細部は面数なりに粗くなります。

### アニメーション（実験的・低ポリ向き）
スキニング（PMXボーン＋VMDモーション）も変換できますが、ddrawの**スループットの壁**で
滑らかには動きません（[docs/FINDINGS.md](docs/FINDINGS.md) 参照）。試す場合は**低ポリ推奨**:

```bash
python tools/model2mdraw.py model.pmx -H 1.6 -f 2500 --vmd dance.vmd -o miku.obj
python tools/model2mdraw.py model.pmx -H 1.6 -f 2500 --vmd test -o miku.obj  # VMDなしのテスト動作
```
```
/mdraw miku 1
/mdraw anim miku_dance    # 全身フレーム方式で再生（アニメ中は密度が大きく落ちる）
/mdraw anim stop          # 静止・高密度に戻す
```

### 表示の向き
設置した瞬間のプレイヤー方向が「正面」になります。向きを変えたいときは
`/mdraw clear` してから見せたい位置に立って置き直してください。

```
plugin/
  MeshSurfaceDraw.cs   Oxide/Carbonプラグイン本体（描画・スキニング）
tools/
  model2mdraw.py       3Dモデル → 拡張OBJ/スキン/アニメ 変換ツール（本体）
  preview_obj.py       拡張OBJの静止プレビュー（プラグインと同じ描画ロジックを再現）

  make_test_models.py  テスト用モデル（球/トーラス）生成

docs/
  FINDINGS.md          検証の詳細な記録（試行錯誤の全履歴）

```
