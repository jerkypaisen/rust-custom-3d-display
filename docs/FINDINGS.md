# 検証の記録（試行錯誤の全履歴）

無改造クライアントに独自3Dモデルを出すために試した手法と、その結果を時系列でまとめる。
同じ轍を踏まないための備忘録。

## 前提: サーバーからクライアントの描画に干渉できる経路

| 経路 | 内容 | 色 | 立体 | 備考 |
|---|---|---|---|---|
| `ddraw.*` | admin宛コンソールコマンドで線/球/箱/テキスト等 | ○ | ○ | 線同士は深度なし(後勝ち)・スループット制限 |
| CUI | サーバー駆動の2D UIオーバーレイ | ○ | ✕ | 画面空間・HUDに重なる |
| サイン/写真立て | PNG(≤1024²)をサーバーからアップロード | ○ | ✕ | 平面・更新はCRC再DLで遅い |
| ネオンサイン | 発光色付きピクセルアート・複数フレーム内蔵 | ○ | ✕ | 平面だが自動フレーム送りあり |
| 彫刻(氷) | ボクセルグリッドをサーバーから彫れる | △ | ○ | 不透明・深度書く・単色/パレット・32³でガタガタ |
| 球エンティティ | sphere.prefab | - | ○ | 半透明・深度を書かない |
| エンティティ移動 | position/rotation更新 | - | ○ | クライアントが滑らかに補間 |

## StringPool の確認（最初の調査）

`StringPool.cs` より、クライアントは**ローカルのアセットバンドルに存在する
プレハブIDしかインスタンス化できない**。任意のカスタムメッシュをネットワーク経由で
送り込んで描画させることは不可能。よって「既存の描画プリミティブを組み合わせて
カスタム形状を表現する」しか道がない。

## ddraw.line スキャンライン充填（採用した主手法）

- サーバーが `SetPlayerFlag(IsAdmin, true)` → `ddraw.line` 送信 → 必要なら戻す
- 三角形を「最長辺へ向かう平行線」で塗り、面を表現
- 画面ピクセル基準の適応密度（近いほど細かい）
- 重ね描き（位相をずらした層）で隙間を補間
- インナーコア（法線方向に少し縮めたコピーを暗い同色で最下層に）で隙間の背景透けを軽減
- 輪郭線はシルエット辺・折れ目辺のみ（全辺に引くと網目状になり破綻）

### つまずきと修正
- **全三角形の辺に輪郭線を引いて網目化** → シルエット/クリース辺のみに限定
- **マテリアル別にポリゴン削減して境界に亀裂** → 全体を一体溶接してから削減・UV/材質は転写
- **透明テクスチャ部分が白い板に** → 透明面を削除＋アトラスのにじみ防止(premultiplied)
- **UV島をまたぐ面が無関係な色を拾う** → 多数派の島に揃える
- **PMXの巻き順をz反転時に未補正で全面裏返り** → 巻き順も反転

## 線同士の深度の検証（決定的だった実験）

検証実験（線の前後関係）: 近い緑線を先に、遠い赤線を後に送って交差表示した。
- **交点が赤** = 後から送った線が常に上 = 線同士は深度判定されない
- これにより「遠→近にソートして送る」必要が判明（画家のアルゴリズム）
- 視点に依存するため、正面観賞前提では「モデル正面の固定視点」を基準にソート

地形・建物に対しては正しく深度テストされる（`zTest=true` で木の後ろに隠れる）ことも確認。

## エンティティによる遮蔽の検証

- 球エンティティ越しの線: 球は半透明で深度を書かず、線が透けた → 遮蔽に使えない
- 氷の彫刻の検証: **氷の彫刻は不透明で深度を書く**。後ろの線がくっきり欠けた
  - `BaseSculpture` の `_grid`(Point3DGrid) にリフレクションでアクセスし、
    サーバーから任意のボクセル形状に彫れることを確認
  - `CleanFloatingIslandsJob` が「y=1層に繋がっていないボクセルを自動削除」するため、
    浮いた島には支柱を下ろして接地させる必要がある
  - → 「モデル形状の氷像を内部に置いて遮蔽コアにする」案を実装したが、
    色が氷単色・ボクセルがガタガタで、線描画との併用は割に合わず断念

## アニメーション（PMXスキン + VMDモーション）

- `model2mdraw.py` で PMX のボーン・頂点ウェイト（BDEF1/2/4, SDEF近似）を `.skin` に、
  VMD のキーフレームを `.anim` に変換（座標系・回転の鏡映を補正）
- プラグインでキーフレーム補間（Slerp）＋サーバー側リニアブレンドスキニング
- `preview_anim.py` で**オフラインでは正しくスキニングできることを確認**（腕が上がる等）

### 部分更新方式 → 破綻
「動く部位だけ毎フレーム再送、静止部位は高密度のまま固定」を実装したが、
**線同士に深度がない**ため、再送した部位（髪・腕）が常に最前面に貼り付き、
背中側に回っても体の前に描かれて破綻。部分更新は原理的に不可能。

### 全身フレーム方式 → スループットの壁
毎フレーム全身をスキニングし、その都度プレイヤー視点で遠→近ソートして全身を再送。
前後関係は正しくなる（深度問題は解決）が:
- 1フレームに送れる線数に上限があり、超えると次フレームが間に合わず点滅・消失
- 実機では低ポリ(2500面)でも1フレーム約2000本が限界で、かなり離れないと見られない
- 線を増やす＝描画に時間がかかり消える時間ができる、というハードリミット

→ ddrawによる滑らかアニメは不可能と結論。

## 到達した結論

「色 + 全方向ボリューム + 高精細 + 滑らかアニメ」を同時に満たすことは
エンジン改造なしには物理的に不可能。各手法は必ず1軸を犠牲にする。
